================================================================================
  建筑工程项目工期预测系统 —— 技术说明
================================================================================

一、系统概述

  系统输入正在施工的水泥厂项目编号，读取该项目的日报数据（每天的人员、机械、
  施工条目），预测下一周所需工人数、机械数及施工活动总量，同时以5个已完工项目
  的甘特图数据为参照，评估当前进度在同类项目中所处的位置。


二、技术架构

  整体数据处理链路分为五层：

  第一层：文本解析
  将731份日报TXT解析为结构化记录。每份日报包含日期、天气、各分包商人数、挖机/
  塔吊/汽车吊等机械数量，以及当天全部施工条目。解析核心为正则表达式——日报为半
  结构化文本，同一文件内既有表格化的"一、施工人员：总人数436人"，也有自由文本
  的施工描述。编号格式的不统一是主要难点：部分日报用"1、2、3"编号，部分用
  "（1）（2）"全角括号编号，部分子条目用"a.b.c."格式。施工条目同时按4大工序
  类型（土建/钢结构/设备安装/装修）和20个工程子项（烧成窑尾/水泥粉磨/煤粉制备
  等）进行分类统计。

  第二层：周级聚合
  将天级数据按周聚合。天级数据波动剧烈（如暴雨导致单日停工，人员数为零），
  按周聚合后日均人数、周总条目数、周降雨天数等指标趋于平滑，信号更清晰。
  甘特图数据同样按周聚合：统计每周完工的任务权重之和作为真实进度标签。

  第三层：特征工程
  将每周转化为206维特征，分四类构造：
  - 当前状态特征：本周人数、机械数量、各工序/子项活动频次；
  - 历史惯性特征：上周、前周、前前周的值以及3周/5周滑动均值，为最强预测信号；
  - 趋势特征：人员增减方向、活动加速或减速；
  - 阶段特征：土建占比高表明处于早期，装修占比上升表明进入后期，以及区域多样性、
    新区域出现速率、Shannon熵等结构信号。
  所有涉及未来的特征均做shift(1)处理，确保训练时不发生信息泄漏。

  第四层：模型训练
  采用XGBoost构建14个独立模型，每个模型仅预测一个目标。不同目标的特征依赖不
  同——人员预测主要依赖历史均值和趋势，机械配置基本不变直接沿用规则，活动频次
  依赖工期位置和子项结构——因此各自调参：人员用平衡参数（max_depth=4）、稳定
  机械用强正则（max_depth=3, alpha=1.0）、波动活动用敏感参数（max_depth=5,
  alpha=0.1）。验证方式为GroupKFold Leave-One-Project-Out，同一项目的周不能
  同时出现在训练和测试中，以防止模型记忆特定项目的模式而丧失泛化能力。

  第五层：预测输出
  输入项目编号，系统自动取最近4周数据、运行特征工程、14个模型各自输出预测值、
  5个已完工甘特图项目提供S曲线基准评估，最终聚合为JSON返回给前端。


三、当前预测能力

  CV交叉验证中正收益的目标（模型预测优于直接用均值猜测）共7个：
    日均施工人员 +36%    最大施工人员 +42%    钢结构活动 +25%
    每日施工条目 +8%     每周施工条目 +8%     设备安装 +4%     装修 +7%

  塔吊和装载机不训练模型，直接沿用上周值（MAE≈0）。

  负收益目标共5个（模型不如直接用均值）：挖机数量、汽车吊数量、机械设备总数、
  土建活动频次、降雨天数。这些目标的预测值不应出现在正式报告中。

  人员预测精度相对较高（日均人员MAE约95人，对800-1200人的项目误差约10%），
  原因在于人员数是连续惯性量，周间变化平缓，两个项目的工人变动模式足够相似。
  塔吊和装载机在项目期间基本不变。
  施工活动趋势方向可判断（上升/下降/平稳），但绝对频次的精度受限。


四、局限性

  1. 无法预测具体的施工内容
  系统可以预测下周设备安装活动的总量，但不能预测具体是哪种工作。如"设备安装
  88次"无法拆分为"窑尾风管浇筑、煤立磨安装、包装机调试、篦冷机焊接"等具体工
  作。根本原因在于施工内容取决于施工组织设计、图纸交付进度、设备到货计划、甲
  供材到场情况及各工序之间的逻辑依赖关系，这些信息不在日报数据中，也不在甘特
  图数据中，属于输入信息的结构性缺失。

  2. 训练数据量不足
  当前仅2个项目共108条周级训练配对。P001的49周和P003的61周处于不同节奏——同
  是第30周，P001的土建活动120次，P003仅5次，同周次活动量差距可达60倍。模型
  无法从如此少量且异质的数据中学到通用的活动演进规律，至少需要10个以上同类型
  项目的完整日报数据才能有效改善。

  3. 活动分类仍不够细
  虽已增加了20个子项区域分类，但每个子项内部仍混杂多种具体工作。以设备安装为
  例，同一标签下实际包含窑尾风管支模浇筑、煤立磨选粉机安装、篦冷机人孔门焊接、
  取料机轨道找正、太极锥组对、辊压机砂浆墩制作等20余种不同的作业。

  4. 相同数字可能对应不同施工阶段
  P003第58周设备安装18次（收尾阶段的零星焊接和轨道找正），第59周骤升至99次
  （主体安装高峰启动——风管支模、非标制作、太极锥组对同时开工）。这一跳跃不
  是正常波动，而是施工内容的质变。模型仅看到数字变化而无法感知内容背后的阶段
  切换，预测值会严重滞后于实际变化。

  5. 仅支持已有日报数据的项目
  系统需要至少2-3周的历史日报数据才能启动预测。对于全新项目或仅有甘特图而无
  日报的项目，无法提供人员和机械预测（只能提供甘特图基准评估）。


五、当前项目状态

  训练数据:
    P001: 327天日报 → 49周, 2025-04 ~ 2026-04
    P003: 404天日报 → 61周, 2025-03 ~ 2026-05

  甘特图基准（5个已完工项目）:
    N791 江西玉山 / N825 肇庆润信 / N812 山东泉兴 / N836 池州中建材 / N686A 梅州皇马

  模型: 14个XGBoost, 206维特征, 分3组参数（stable/personnel/volatile）
  验证: GroupKFold Leave-One-Project-Out 2折
  数据: 天→周聚合, 本周→下周配对训练
  输出: JSON接口（前端调用）+ 文本报告


六、总结

  系统定位：
    资源预判：下周需要多少人、多少机械，当前进度与同类项目相比是快是慢。
    非施工计划：不能替代施工计划，无法指定具体施工内容。

  短期可改进（无需新数据）：
    人员预测输出置信区间而非点值；砍掉负收益目标的报告展示；甘特图做同阶段
    任务推荐。

  长期可改进（需数据积累）：
    积累10个以上同类型项目的完整日报后，活动预测精度将有质的提升。在此之前，
    "下周需要多少资源"比"下周具体做什么"更适合作为模型的输出目标。
