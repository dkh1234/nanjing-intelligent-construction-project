# 建筑工程项目工期预测系统 —— 代码实现梳理与技术优化方向

---

## 一、项目架构总览

### 完整处理链路

```
甘特图.xlsx（5份） → process_gantt.py → unified_tasks.csv → weekly_gantt.csv ──┐
日报.txt（731份）  → process_daily.py → weekly_daily.csv ────────────────────┤
                                                                             ├→ train_forecast.py → models/*.pkl（14个模型）
                                                                             ├→ benchmark.py → S曲线基准
                                                                             └→ predict_api.py → JSON（前端调用）
```

### 技术栈

Python 3.13 + pandas + numpy + xgboost + scikit-learn + joblib + openpyxl，共 6 个核心包，无深度学习框架，无 GPU 需求。

## 二、逐模块代码分析

### 2.1 process_gantt.py —— 甘特图 → 周级训练集

**输入**：5 份 WBS 任务 Excel 文件  
**输出**：`unified_tasks.csv`（3520 条清洗后任务）、`weekly_gantt.csv`（456 周，含真实进度标签）

**核心流程**：

| 步骤 | 方法 | 关键操作 |
|------|------|---------|
| 加载合并 | `pd.read_excel` + openpyxl | 按位置统一列名（兼容有无"计划开始/结束"两种列结构），只保留叶子任务（type=作业），加 project_code |
| 清洗日期 | 日期比较 + 推算 | 修复 N825 的 2001-01-17 脏数据；137 条缺失 actual_end 用 start+dur 估算 |
| 权重归一化 | 项目内除以总和 | 每个项目内部 weight_norm = weight_pct / sum(weight_pct) |
| 周级展开 | `pd.date_range(W-MON)` | 每周统计：完工任务权重之和（标签）、活跃任务数、新开工数、延期数 |
| 裁剪零段 | 首尾空周检测 | 前导超过 24 周零段保留最后 4 周；无活跃任务→删除 |
| 衍生特征 | 统计聚合 | avg_task_dur / total_remaining_work / delay_ratio_to_date / completed_this_week |

**关键设计决策**：
- 进度标签来自甘特图的任务权重完成率，**不是伪造的**——这是唯一有真实进度标签的数据源
- 权重复用甘特图内部的原始权重体系，确保跨项目可比

---

### 2.2 process_daily.py —— 日报 TXT → 周级特征

**输入**：按项目分目录存放的每日日报 TXT  
**输出**：`weekly_daily.csv`（P001=49 周 + P003=61 周 = 110 周）

**核心流程**：

| 步骤 | 方法 | 关键操作 |
|------|------|---------|
| 单份日报解析 | `parse_one_report()` | 正则提取日期、气温、工人数（优先"总人数"模式）、分包商数、各机械数量 |
| 施工条目提取 | 正则 `\d+[\.\、）)]` + `（\d+）` + `[a-z][\.\)]` | 匹配三种编号格式，统计条目数 |
| 工序分类 | `_categorize_work()` | 4 类关键词匹配：土建 / 钢结构 / 设备安装 / 装修 |
| 子项分类 | `_classify_sub_project()` | 20 个工程子项区域：烧成窑尾 / 水泥粉磨 / 煤粉制备 / 辅料堆场... |
| 周级聚合 | 7 天汇总 | 日均人数、周降雨天数、四类工序频次、20 子项活跃度 |
| 签名特征 | Counter + 熵计算 | 子项多样性 sub_diversity、新区域速率 new_sub_areas、Shannon 熵 sub_entropy、累计区域数 cum_sub_areas |

**关键设计决策**：
- 日报的总体进度是未知的——没有整体完成百分比。因此日报不提供进度标签，只提供人员 / 机械 / 天气 / 施工活动等**执行层特征**

---

### 2.3 train_forecast.py —— 多目标预测模型训练

**输入**：`weekly_daily.csv`（周级特征）  
**输出**：`models/forecast/*.pkl`（14 个 XGBoost 模型）+ `feature_columns.txt`（206 维特征列表）

**核心流程**：

| 步骤 | 方法 | 关键操作 |
|------|------|---------|
| 特征工程 | `build_features()` + `build_training_pairs()` | 构造 206 维特征，分四类：历史惯性 / 当前状态 / 变化趋势 / 阶段信号 |
| 配对训练 | 第 N 周特征 → 第 N+1 周目标 | 每个相邻周配对为一条训练样本（108 条） |
| 分目标训练 | XGBoost × 14 个独立模型 | 3 组参数：stable（强正则）/ personnel（平衡）/ volatile（弱正则） |
| 交叉验证 | GroupKFold(n_splits=2) | P001↔P003 互验，防止项目间信息泄漏 |
| 评估 | MAE 对比基准 | 基准 = 直接用训练集均值做预测的 MAE；改善率 = (基准 MAE - 模型 MAE) / 基准 MAE |

**14 个预测目标**：

| 类别 | 目标 | 参数组 |
|------|------|:-----:|
| 人员（2） | 日均施工人员、最大施工人员 | personnel |
| 机械（3） | 挖机数量、汽车吊数量、机械设备总数 | stable |
| 机械规则（2） | 装载机数量、塔吊数量 | 不训练，沿用上周值 |
| 活动（6） | 每日/周施工条目数、土建/钢结构/设备安装/装修活动 | volatile |
| 天气（1） | 降雨天数 | volatile |

**206 维特征分类**：

| 类别 | 构成 | 信号强度 |
|------|------|:------:|
| 历史惯性 | lag1/2/3 + ma3/ma5（各核心指标） | 最强——人员预测中 avg_workers_lag1 排第一 |
| 当前状态 | 本周人数、机械、工序频次、子项活跃度 | 参照基线 |
| 变化趋势 | workers_trend、activity_trend、equip_trend、change_rate | 中——判断阶段切换 |
| 阶段信号 | stage_early/mid/late_score、stage_transition、sub_entropy、区域占比 | 弱——但数据多了会更强 |

**关键约束**：所有 lag/ma 用 shift(1) 构造，确保训练时不会用到未来信息。

---

### 2.4 benchmark.py —— 甘特图 S 曲线基准评估

**输入**：甘特图周级数据 + 当前项目周级数据  
**输出**：进度评估（"快 / 正常 / 慢"）

**核心流程**：
1. **S 曲线构建**：5 个已完工甘特图项目按相对位置（0%-100%）插值对齐，计算每个位置的中位数 / P25 / P75
2. **总工期估算**：当前周次 × 阶段因子（活动量占比推算） vs 甘特图项目平均工期
3. **基准对比**：计算当前项目相对位置，匹配最近的基准点，输出同位置已完工项目的平均完成百分比

---

### 2.5 predict_api.py —— JSON 预测接口

**输入**：项目编号 "P003"  
**输出**：JSON（personnel / equipment / activity / weather / benchmark）

**内部流程**：
1. 取该项目最近 4 周数据 → `build_features()` → 206 维特征
2. 14 个模型各出一个预测值
3. 后处理规则裁剪（人员周环比不超过 ±30%、活动不超过 ±50%、恒定设备沿用上周值）
4. benchmark.py 跑基准评估
5. 组装 JSON：每个预测值带 value / unit / range / rule_based 等字段

**前端调用方式**：
```bash
python src/predict_api.py --project P003           # 输出 JSON
python src/predict_api.py --list-projects           # 列出可用项目
```

## 三、技术优化方向

### 3.1 数据处理层：正则解析是最大单点故障

**现状问题**：
1. **编号格式不可枚举**。目前支持 `1、` / `（1）` / `a.` 三种。海螺粉磨站 DOCX 里出现了 `（1）` 嵌套 `a.` 的结构，未来不同施工单位的编号习惯千差万别（`1)` `1.1` `[1]`...），正则补丁会越打越多
2. **工序关键词会过时**。`WORK_CATEGORIES` 和 `SUB_PROJECTS` 两套字典是针对现有水泥厂项目手调的。每来一个新项目类型就得加一次关键词
3. **温度和人数存在边缘格式**。"气温3℃～21℃" 修了一次，后面可能出现 "3-21℃"、"3到21度" 等变体。人数的 "总人数" 优先规则也是事后补的——不同项目格式未必一致

**优化方向**：
- 近期：将关键词从硬编码改为配置文件（YAML / JSON），新增项目类型时只改配置不改代码
- 中长期：考虑用小模型（BERT 级别的 NER）做信息抽取——人员、数量、设备名、工序名，训练几百条标注数据即可覆盖大部分变体

---

### 3.2 特征工程层：冷启动与膨胀

**现状问题**：
1. **冷启动**：shift(1) 构造的 lag 特征要求至少 3-4 周历史数据，新项目前几周不具备预测条件
2. **特征膨胀**：当前 20 个子项区域 × 每个 6 个 lag/ma 衍生物 = 120 维。项目多了以后更多子项出现，维数会进一步增加。XGBoost 天然处理高维稀疏特征，这本身不是性能问题，但需要跟踪是否出现无用特征

**优化方向**：
- 冷启动：用所有历史项目的"同阶段均值"填补新项目前几周的缺失特征
- 特征选择：定期做 importance 排序，砍掉长期排在尾部的特征

---

### 3.3 模型训练层：三个待改进点

**① 14 个独立模型丢失了目标间的关联信息**

人员预测和活动预测是相关的——人多的时候活动也多。14 个独立模型各自学各自的，不知道 "人员预测 +10% 意味着活动预测也应该增加"。

优化方向：项目数据达到 10+ 以后，可以考虑多输出回归（sklearn 的 MultiOutputRegressor 包装 XGBoost，或换成 LightGBM 的多输出模式），或至少加一个残差关联层——用人员预测的残差作为活动预测的额外特征。

**② 基准太弱**

当前用"猜均值"作为基线是对起步阶段的诚实应对。但随着数据增多，可以上更强的基线：

- "上周值"基线：`predict(x) = 上周的实际值`——对人员预测，这个基线可能比 XGBoost 还强
- ARIMA / 简单时序模型：比猜均值强，比 XGBoost 简单
- 行业经验规则：如"项目中期人员 = 800-1000 人"，作为领域常识基线

如果 XGBoost 连"上周值"都打不过，说明特征工程没抓到增量信息。

**③ 缺少项目类型感知**

P001/P003 是熟料线，P004（海螺）是粉磨站，施工模式不同——但当前模型不做区分。

优化方向：5+ 项目以后，尝试**按项目类型分组建模**（熟料线一组、粉磨站一组、矿山一组），或加"项目类型"作为分类特征。

---

### 3.4 预测输出层：缺少分布外检测

**现状问题**：当前模型对新项目类型不做任何告警，直接输出预测值。但 P004（海螺粉磨站）的施工特征分布可能与 P001/P003（熟料线）不同——模型未曾见过，预测可靠性会下降。

**优化方向**：加一个简单的特征分布距离检测。用训练集的均值/方差做标准化，对新项目计算马氏距离。如果距离超过 2-3 个标准差，在输出中标记："⚠ 该项目类型与训练数据差异较大，预测仅供参考"。

---

### 3.5 甘特图基准层：信息利用不充分

**现状问题**：当前 benchmark 只输出一句话："同位置应完成 69%"。5 个已完工项目包含的数据远不止于此。

**优化方向**（5+ 项目后）：
- **分项目类型做基准**：熟料线 vs 粉磨站 vs 矿山，各自建立 S 曲线
- **阶段任务推荐**：匹配最相似历史项目，列出它在同阶段做了哪些类型任务
- **进度偏差预警**：当前进度低于 P25 时自动告警

---

### 3.6 工程架构：版本管理缺失

**现状问题**：
- `weekly_daily.csv` 被 `process_daily.py` 直接覆盖写入，没有版本号，无法追溯是哪次跑的
- 模型文件名靠时间戳区分，但没有记录对应的特征列、训练数据版本、超参
- 新日报进来后需手动三步操作：`process_daily` → `train_forecast` → 预测

**优化方向**：
1. 数据版本号：`weekly_daily.csv` 文件名带日期或加 meta 列
2. 模型元数据：保存模型同时写一个 JSON（特征列表、训练样本数、CV 分数、数据版本）
3. 增量处理：`process_daily.py` 支持只处理新增的 TXT，不全量重跑

## 四、数据规模现状

| 层级 | 数据量 |
|------|:----:|
| 日报 TXT | P001=327 天 + P003=404 天 = **731 份** |
| 甘特图任务 | 5 个项目 × 平均~700 个叶子任务 = **3,520 个** |
| 周级训练配对 | **108 条**（第 N 周→第 N+1 周） |
| 特征维数 | **206 维**（含 lag/ma/趋势/阶段/子项衍生） |
| 预测目标 | **14 个**（12 个训练 + 2 个规则直出） |
| 正收益目标 | **7/12**（人员 2 + 活动 5，改善幅度 4%-32%） |

## 五、优化优先级总览

| 优化项 | 时机 | 理由 |
|--------|:----:|------|
| 正则→配置文件 | 下个新项目前 | 避免每次加关键词改代码 |
| 分布外检测 | 现在可做 | P004 粉磨站预测已需要 |
| 更强的基线（上周值） | 下轮训练时 | 对标才知道 ML 真实价值 |
| 模型元数据 | 下个新项目前 | 否则不知道模型用哪些数据训的 |
| 增量处理 | 下个新项目前 | 避免每次全量重跑 |
| 分项目类型建模 | 5+ 项目 | 熟料线/粉磨站/矿山分开 |
| 阶段任务推荐 | 5+ 项目 | 甘特图信息利用 |
| 多输出建模 | 10+ 项目 | 数据够多才能学到目标间关联 |
| NER 替代正则 | 10+ 项目 | 标注数据积累 |
