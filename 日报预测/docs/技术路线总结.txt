基准模型  即不作任何特征学习，永远猜整个训练集的目标均值

基准模型（Baseline）：我们把整个训练集所有周的目标值（比如"日均施工人员"）的平均值作为预测值——不管输入什么，永远猜这个数。这个"只会猜均值"的模型就是基准模型。

比如日均施工人员，所有周的历史均值大约是 800 人。基准模型就永远预测"下周 800 人"。它的 MAE = 401 人——平均每次猜错 401 人。

XGBoost 模型
根据本周的特征（人数趋势、机械配置、工序活跃度、降雨等 206 个维度）来预测下周的具体值。

跑完 2 折 GroupKFold（P001↔P003 互验），XGBoost 的 MAE = 318 人。

定义MAE（核心概念）：

全称：平均绝对误差（Mean Absolute Error）。

定义：预测值与实际观测值之间绝对误差的平均值。

一、起点：手头有什么资料

  日报数据（2个项目）：
    - P001：327天 .txt 日报，每天包含日期、天气、各分包商人数、挖机/塔吊/汽车吊
      等机械数量、按编号列出的施工条目（如"窑尾风管支模浇筑"）
    - P003：404天 .txt 日报，格式同上，但文字风格与P001略有差异

  甘特图数据（5个项目）：
    - 5个已完工水泥厂项目的施工计划xlsx，每个文件是一个WBS任务树，记录了每个
      叶子任务的任务名称、计划工期、实际开始/结束日期、权重%、完成百分比

  两类数据的特点：
    - 日报是"执行记录"：每天实际发生了什么，有人员/机械/天气/施工条目，但没
      有整体进度百分比
    - 甘特图是"计划vs实际"：知道每个任务从哪天到哪天、占多少权重、做完了没有，
      但没有天气/人力/机械等执行细节


二、关键发现：初始方案的标签是伪造的

  项目最初的dataClean/src/data_parser.py中，累计进度的计算方式是：
    cumulative_progress = active_areas * 2.0
  即"当天日报写了几个施工条目×2"当作当天累计进度%。这跟实际工程进度没有任何
  关系——日报写得多不代表工程完成得多。

  这意味着：如果沿用"预测进度增量"作为目标，整个训练链路的标签都是虚假的。


三、为什么改为按周预测

  最初考虑按天预测，但做了三份诊断后决定改为按周：

  1. 天级数据噪声大：单日暴雨停工致人员归零、次日加人赶工翻倍，波动掩盖真实模式。
  2. 周级聚合平滑噪声：7天均值让信号更清晰。
  3. 管理需求匹配：工地按周汇报进度，周报比日报更有决策意义。

  代价：样本量从~730天缩至~108周，但信号质量提升远大于样本量损失。


四、为什么改为"预测资源"而非"预测进度"

  已知日报的进度标签是伪造的，摆在面前有两条路：
    路线A：用甘特图的权重完成率给日报项目构造伪标签，联合训练继续预测进度。
    路线B：放弃进度预测，改为预测日报本身就有真实记录的东西——人员、机械、活动。

  路线A试了：通过追踪日报中施工活动签名的首次出现时间构造伪进度，但很快暴露
  缺陷——所有签名在项目中期已出现完毕（"所有施工类型都出现过了"），伪进度在
  中后期失真。

  最终选择路线B：预测目标从"一个抽象的进度百分数"变成"下周需要多少资源"——
  人员数、机械数、活动频次。这些标签在日报中有真实记录，不需要伪造。


五、模型选择：为什么用XGBoost

  候选模型对比：
    线性回归：捕捉不了非线性关系（项目活动有阶段性突变），R2为-3.57，直接排除。
    Random Forest：稳定性好但精度低于XGBoost，CV R2仅0.01。
    XGBoost：CV MAE=0.0076，R2=0.37，三项中表现最优。

  XGBoost在此场景下的优势：
    1. 自动处理缺失值（日报部分字段偶有空缺）
    2. 特征重要性可解释（知道"上周人数"和"烧成窑尾活动量"哪个更重要）
    3. 小数据下通过正则化（reg_alpha/reg_lambda）控制过拟合
    4. 对tablular特征天然友好，无需归一化

  为什么不拆成14个单独模型而非一个多输出模型：
    不同目标的最佳特征组合不同——人员主要靠历史均值和lag，机械配置几乎不变，
    活动依赖工期位置和区域结构。拆开后各自调参、各自评估，一个差不影响其他。


六、为什么用GroupKFold Leave-One-Project-Out

  将数据划分为 K 个大小相等的子集（折）。迭代训练 K-1 折，并在剩余的 1 折上验证。重复 K 次

  常规KFold会将数据随机打散，同一项目的不同周可能分散在训练集和测试集中。这
  会导致数据泄漏：模型记住了"P003第30周的工作模式"，在测试集上见到P003的其他
  周时表现虚高，但面对新项目时崩溃。

  GroupKFold以项目为分组边界，确保同一项目的所有周要么全在训练集、要么全在测
  试集。2个项目做2折，每折用1个项目训练、1个项目验证——虽然折数少，但保证了
  跨项目泛化能力的评估是诚实的。


七、甘特图的定位：从训练数据变为基准参考

  甘特图最初被考虑作为训练数据（5个已完工项目有真实进度标签），但甘特图没有
  日报那些天气/人力/机械字段，特征空间与日报不匹配，无法联合训练。

  最终定位为"S曲线基准"：将5个已完工项目按相对位置插值对齐，计算每个位置的中
  位数/P25/P75完成百分比。当前日报项目输入"我在第XX周/占总工期YY%"，基准告诉
  你"同位置项目通常完成ZZ%"。这是日报系统本身提供不了的视角。


八、特征工程的设计逻辑

特征工程，就是把原始数据变成模型能听懂、且更容易学习的“专业语言”。

  206维特征按构造逻辑分为四类：

  历史惯性（最强信号）：上周、前周、前前周的值 + 3周/5周滑动均值。在人员预测
  中，avg_workers_lag1的重要性长期排第一或第二。这符合直觉——施工现场的
  资源配置有连续性，不会今天1000人明天突然变成200人。

  当前状态：本周实际人数、机械数、各工序频次、各子项活跃度。这些是滞后特征的
  参照系。

  变化趋势：人员变化方向、活动增减幅度。用来判断是否进入阶段切换。

  阶段信号：土建占比高→早期，装修占比高→后期，区域多样性熵→反映施工作业面
  的广度。这些是日报特有的构造特征，从工序分布反推项目阶段。

  所有特征在构造时做shift(1)处理——用昨天的数据预测明天的值，确保训练时不会
  意外用未来信息。


九、预测输出的内容与边界

  输出六大板块（当前版本）：
    一、人员预测：日均/最大施工人员 + 置信区间
    二、机械预测：挖机/汽车吊/装载机/塔吊/设备总数（恒定设备标[沿用上周]）
    三、活动预测：每日/每周条目数 + 四类工序频次 + 子项区域活跃度
    四、天气预测：预计降雨天数（实际无气象输入，不可靠）
    五、基准评估：当前周次/估算总工期/相对位置/同位置已完工项目完成百分比/
        活动趋势/人员稳定性/综合评估

  能做到的：
    - 下周需要多少人：日均人员MAE约95人（误差约10%），输出区间而非点值
    - 塔吊/装载机：沿用上周值，MAE≈0
    - 活动趋势方向（上升/下降）

  做不到的：
    - 具体哪个区域做什么活：施工内容取决于工地实际安排，不在数据中
    - 工序绝对频次的精确值：2个项目训练数据不足以覆盖不同阶段的模式差异
    - 天气：无气象数据输入


十、数据流全景

  原始数据 → 文本解析 → 周级聚合 → 特征工程 → 模型训练 → 预测输出

  日报TXT (731份) → 天级结构化记录 → weekly_daily.csv (110周, 有真实标签)
                                                  ↓
  甘特图xlsx (5份) → 任务表(3520条) → weekly_gantt.csv (456周, 有真实标签)
                                                  ↓
                                    特征工程 → 206维特征
                                                  ↓
                                    XGBoost ×14 → 本周→下周配对训练
                                                  ↓
                                    predict_api.py → JSON
                                                  ↓
                                    benchmark.py → S曲线基准评估


十一、演进过程中的关键决策点

  1. 发现日报进度标签是伪造的 → 放弃"预测进度"，改为"预测资源"
  2. 天→周聚合 → 牺牲样本量换取信号质量
  3. 甘特图从训练数据退为基准参考 → 各司其职，不强行融合
  4. 多目标拆分为14个单模型 → 各自评估、各自调参、各自判断可用性
  5. 子项区域分类扩展（4类→20类） → 负收益目标从9个降至5个
  6. 后处理规则引入（恒定设备、环比裁剪） → 压掉不可能的输出值


十二、当前不足

  1. 训练数据仅2个项目108条配对：CV只做2折，泛化能力难以精确评估。新项目
     进来后可能分布偏移导致精度下降。
  2. 活动预测精度受限于项目数量：7个正收益目标中，活动的改善幅度较小（+4%~+8%），
     根本限制在数据量。
  3. 没有天气真实输入：降雨天数预测完全靠猜。
  4. 预测输出中仍有5个负收益目标（挖机/汽车吊/设备总数/土建/降雨），这些不应
     出现在正式报告中。
  5. 无法处理全新项目：需要至少2-3周日报数据才能启动预测。
